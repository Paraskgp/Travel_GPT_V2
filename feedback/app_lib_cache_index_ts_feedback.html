app/lib/cache/index.ts Feedback

Target file: app/lib/cache/index.ts Feedback type: Cache architecture and quality risk

Elephant In The Room

The cache makes generated quality durable, but it does not distinguish verified facts from generated prose. A bad board can become sticky, and prompt-hash invalidation only catches prompt changes, not source-quality changes or validation failures.

Major Feedback

Suggested Implementation Direction

  • Add schema_version, model, provider, source_inventory_hash, and validation_status to cache entries.
  • Add a separate cache key for candidate_inventory_{source_hash}.
  • Add an admin/status view that surfaces "cached but failed validation" and "cached with low source coverage".
  • Do not serve cached boards to planning if their validation status is failed or unknown.

Principal Engineer Implementation Notes

The cache should preserve trust metadata, not just generated output. Otherwise bad facts become durable and hard to diagnose.

Implementation-Ready Checklist

Acceptance Criteria

  • A cached board can be traced to the facts and prompt version that produced it.
  • Schema changes do not accidentally serve stale boards as current.
  • The planner can refuse cached boards that failed hard validation.

Needs Product Decision

  • Decide whether old caches are auto-invalidated or grandfathered. Recommendation: invalidate old boards for planning, but allow display with a warning if needed.

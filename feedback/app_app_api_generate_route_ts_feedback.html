app/app/api/generate/route.ts Feedback

Target file: app/app/api/generate/route.ts Feedback type: Board generation architecture

Elephant In The Room

This route still lets the LLM create the board as the primary source of truth. Search grounding is present, but it is injected as context instead of enforced as a data contract.

Major Feedback

Suggested Implementation Direction

  • Create a CandidateExperience inventory type and make theme cards reference candidates by ID.
  • Add source_refs, verification_status, confidence, and canonical_place_id to generated card data.
  • Add validateBoard(board, candidateInventory) before cacheWrite.
  • If validation fails, return a partial board with warnings or regenerate only the failed cards.
  • Cache candidate inventory separately from final card prose.

Principal Engineer Implementation Notes

This route is fixable by splitting it into discovery, verification, selection, writing, and validation. Do not try to solve this by adding one more prompt instruction.

Implementation-Ready Checklist

Acceptance Criteria

  • A hallucinated card cannot be cached as verified.
  • Every generated card can be traced back to a source-backed candidate or explicitly marked unverified.
  • The route response includes board-level validation warnings when quality is degraded.
  • Existing UI can still render the board while richer metadata is phased in.

Needs Product Decision

  • Decide whether /api/generate should return partial boards with warnings when validation fails, or fail closed for high-risk destinations. Recommendation: partial board is acceptable, but automatic itinerary planning should be blocked for failed hard validation.

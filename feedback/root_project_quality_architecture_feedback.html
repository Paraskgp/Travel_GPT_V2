Root Project Quality Architecture Feedback

Target area: Cross-file system architecture Primary files to read with this feedback: PRODUCT_SPEC.md, app/PRODUCT_SPEC.md, ARCHITECTURE.md, app/app/api/generate/route.ts, app/app/api/plan/route.ts, app/lib/types.ts

Elephant In The Room

TravelGPT is treating quality as a prompt quality problem, but the larger gap is architectural: the system does not yet have a durable, structured model of verified travel facts, constraints, and scheduling rules.

Right now, the pipeline asks LLM stages to synthesize, remember, infer, verify, route, schedule, critique, and repair. Some search grounding has been added, but the outputs are still mostly prose and loose strings. The itinerary planner is therefore operating on "cards that sound right", not on a normalized travel knowledge layer that can be checked and enforced.

Major Feedback

Suggested Fix Sequence

  1. Extend app/lib/types.ts with structured ExperienceFact, AccessConstraint, ScheduleConstraint, and SourceRef concepts.
  2. Change app/app/api/generate/route.ts so search grounding produces a verified candidate inventory before theme writing.
  3. Add a validation stage before board cache write that rejects or flags cards without adequate source coverage.
  4. Change app/app/api/plan/route.ts so planning begins with deterministic constraint extraction and anchor placement.
  5. Let prompts improve style and judgment only after the data model and gates exist.

Success Criteria

  • A fabricated place cannot enter the board unless a source verifier fails.
  • A restricted-access activity cannot be scheduled with an impossible travel mode.
  • A family_young itinerary cannot exceed activity density limits unless the user explicitly overrides.
  • Eval failures point to a broken data contract or gate, not only "try another prompt wording."

Principal Engineer Implementation Notes

This is fixable, but only if the next change creates enforceable data contracts. Treat prompts as writers over facts, not as the reliability layer.

Implementation-Ready Checklist

Acceptance Criteria

  • Every board card has a source-backed candidate ID or an explicit verification_status: "unverified" that blocks automatic scheduling.
  • Hard logistics constraints are represented as data and checked in code.
  • The itinerary route can explain why an item was scheduled, skipped, or blocked.
  • A prompt-only change is no longer the only way to fix shuttle, timing, density, or hallucination failures.

Needs Product Decision

  • Decide whether unverified-but-interesting experiences can be shown on the board as exploratory content, and whether they are blocked from itinerary scheduling by default.
  • Decide the user-facing language for uncertainty: warning banner, confidence chip, or hidden until verified.

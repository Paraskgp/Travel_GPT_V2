app/lib/places/client.ts Feedback

Target file: app/lib/places/client.ts Feedback type: Places verification and enrichment architecture

Elephant In The Room

Places is currently used as a presentation enrichment layer after board generation. It should also be part of the trust layer that decides whether a card is safe to map and schedule.

Major Feedback

Suggested Implementation Direction

  • Change enrichExperience to return { enrichment, verification }.
  • Add normalized-name matching and destination-bounds checks.
  • Persist accepted place_id and coordinates back onto the board or candidate inventory.
  • Reject or warn on low-confidence matches instead of silently returning the first result.

Principal Engineer Implementation Notes

Places should become a verifier as well as a photo provider. Start with security and match confidence.

Implementation-Ready Checklist

Acceptance Criteria

  • No secret key is present in source code.
  • The first Places result is not accepted blindly.
  • A low-confidence map match does not silently become an itinerary anchor.
  • Planning can access coordinates before or during route generation.

Needs Product Decision

  • Decide whether low-confidence Places matches should hide the card from the map, block scheduling, or both. Recommendation: hide map pin and block automatic scheduling until verified.

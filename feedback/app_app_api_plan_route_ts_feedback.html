app/app/api/plan/route.ts Feedback

Target file: app/app/api/plan/route.ts Feedback type: Itinerary planning architecture

Elephant In The Room

The planner is asked to solve constraints through LLM reasoning after receiving mostly prose-like board data. That is why timing, shuttle rules, family density, and recovery time keep slipping. The architecture needs deterministic constraint extraction and validation before and after the LLM draft.

Major Feedback

Suggested Implementation Direction

  • Add extractSchedulingConstraints(board, preferences, dates) before draft generation.
  • Add validateItinerary(itinerary, constraints, forced_ids, skipped_ids) after every LLM planning stage.
  • Fail closed for hard violations like restricted access, forbidden experiences, or departure-day violations.
  • Return unscheduled_experiences and validation_warnings in PlanResponse.
  • Use deterministic anchors for time-locked experiences, then let the LLM compose the human-readable day narrative around those anchors.

Principal Engineer Implementation Notes

This route should become constraint-first and composition-second. Flexible itineraries are good; impossible itineraries are not.

Implementation-Ready Checklist

Acceptance Criteria

  • The planner can still make flexible choices among soft preferences, but hard constraints are never left to prose reasoning alone.
  • A reviewer failure does not turn an invalid Pass 1 draft into a trusted itinerary.
  • The API tells the UI and eval scripts what was skipped and why.
  • User overrides are respected and checked after generation.

Needs Product Decision

  • Decide the planning mode: auto-plan from the whole board, plan from liked experiences only, or auto-plan then let user pin/skip. Recommendation: for the current product, auto-plan from the board is fine if the UI says so and exposes skipped/selected reasoning.
